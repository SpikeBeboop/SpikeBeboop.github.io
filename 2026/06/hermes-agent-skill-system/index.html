适用对象：想理解、编写、安装、调用、维护 Hermes Agent Skills 的用户或开发者。

依据来源：Hermes 官方文档 Skills System、Creating Skills，以及本机 Hermes 源码中的 tools/skills_tool.py、tools/skill_manager_tool.py、agent/skill_commands.py、agent/prompt_builder.py、agent/skill_utils.py、agent/skill_preprocessing.py。

核心结论：Skill 是 Hermes 的“按需加载的程序性知识”，不是工具本身；它通过 Markdown + YAML frontmatter 描述何时使用、怎么做、有哪些坑、如何验证，并可附带 references/templates/scripts/assets 等支持文件。

1. Skill 是什么#

Hermes Agent 的 Skill 是一种按需加载的知识包，用来让 agent 在特定任务中获得更专业、更稳定的操作流程。

它通常包含：

  • 触发条件：什么时候应该加载这个 skill。
  • 操作流程：agent 应该按什么步骤完成任务。
  • 工具/命令用法：CLI、API、脚本、Hermes 工具调用建议。
  • 坑点与失败处理：常见错误、环境限制、替代路径。
  • 验证标准：任务完成前如何确认结果可靠。
  • 支持文件：参考文档、模板、脚本、资产文件。

Skill 的本质是“过程性记忆 / procedural memory”。

它和普通 memory 的区别：

类型 用途 典型内容 是否常驻上下文
Memory / User Profile 持久事实、偏好、环境约定 用户偏好、项目路径、稳定事实 是，通常注入每轮
Skill 可复用工作流和专业流程 如何调研论文、如何发布 PR、如何用某工具 否，按需加载
Tool 可执行能力 读文件、跑命令、浏览器、发消息 以工具 schema 形式暴露

一句话：

  • Memory 记“事实”。
  • Skill 记“怎么做”。
  • Tool 负责“执行动作”。

2. Skill 适合解决什么问题#

适合做成 Skill 的场景：

  1. 任务能用“说明 + 现有工具 + shell/API/脚本”表达。
  2. 需要稳定复用同一套流程。
  3. 任务流程比一句提示复杂，容易遗漏步骤或坑点。
  4. agent 曾经通过多轮尝试找到可靠做法。
  5. 用户明确纠正过某类任务的做法。
  6. 需要让后续 agent 以一致标准执行。

典型例子：

  • GitHub PR 工作流。
  • 代码审查流程。
  • 大型源码学习流程。
  • Excel 数据分析流程。
  • 调研报告撰写规范。
  • 特定 CLI/API 的使用手册。
  • 某网站自动化操作经验。
  • 图像生成/前端设计质量标准。

不适合做成 Skill 的场景：

场景 更适合
需要精确、强约束、每次一致执行的计算逻辑 Tool 或脚本
需要处理流式事件、二进制协议、复杂认证 Tool / 插件
只是一个短期任务状态 不保存；留在当前会话即可
一周后会过期的进度、PR号、commit SHA 不写 memory，也不写 skill
单次临时说明 当前对话即可

3. Skill 总体架构#

Hermes Skill 系统可以分成 7 层：

用户请求
  ↓
系统提示中的 Skill 索引（只含 name/description/category）
  ↓
模型判断相关 skill
  ↓
skill_view(name) 加载完整 SKILL.md
  ↓
需要时 skill_view(name, file_path) 加载支持文件
  ↓
agent 按 Skill 指令调用 terminal/file/web/browser 等工具执行任务
  ↓
如果发现新流程或修正旧流程，用 skill_manage 维护 skill

关键原则是 progressive disclosure（渐进披露）：

层级 入口 加载内容 目的
Level 0 系统提示 / skills_list 名称、描述、分类 让模型知道有哪些 skill
Level 1 skill_view(name) 完整 SKILL.md、元数据、支持文件列表 任务相关时加载完整流程
Level 2 skill_view(name, file_path) 单个 references/templates/scripts/assets 文件 只在需要时加载大文件/细节

这样能避免把所有 skill 全文塞进上下文，减少 token 消耗。

4. Skill 的存放位置#

4.1 本地主目录#

当前 Hermes 文档定义的主 skill 目录是：

~/.hermes/skills/

这是运行时的 single source of truth：

  • 内置 bundled skills 安装/更新后会同步到这里。
  • hub 安装的 skills 会放到这里。
  • agent 通过 skill_manage(action='create') 创建的新 skill 会放到这里。
  • agent 可通过 skill_manage 修改这里的 skill。

示例结构：

~/.hermes/skills/
├── mlops/
│   └── axolotl/
│       ├── SKILL.md
│       ├── references/
│       ├── templates/
│       ├── scripts/
│       └── assets/
├── github/
│   └── github-workflows/
│       └── SKILL.md
├── software-development/
│   └── test-driven-development/
│       └── SKILL.md
├── .hub/
│   ├── lock.json
│   ├── quarantine/
│   └── audit.log
└── .bundled_manifest

4.2 Profile 隔离#

Hermes profile 有独立的 skills 目录。

默认 profile：

~/.hermes/skills/

命名 profile：

~/.hermes/profiles/<profile-name>/skills/

影响：

  • 当前会话只加载 active profile 的 skills。
  • 修改另一个 profile 的 skills 需要显式切换 profile 或使用带 cross-profile 权限的文件工具。
  • skill_manage 默认只操作当前 active profile。

4.3 外部 Skill 目录 external_dirs#

Hermes 支持额外扫描外部 skill 目录。

配置位置：

skills:
  external_dirs:
    - ~/.agents/skills
    - /home/shared/team-skills
    - ${SKILLS_REPO}/skills

行为：

  • 本地 ~/.hermes/skills/ 优先。
  • external dirs 也会出现在系统 prompt、skills_list、skill_view、slash command 中。
  • 如果同名，local 版本优先；某些加载路径遇到歧义会拒绝猜测，要求用分类路径明确指定。
  • skill_manage 修改已存在 skill 时，会改它实际所在位置，包括 external_dirs 中的可写 skill。
  • 新建 skill 默认写入 local skills dir，不写 external_dirs。

4.4 Bundled / Optional / Hub / Plugin Skills#

类型 来源 安装/发现方式 信任级别
bundled skills Hermes 仓库 skills/ 安装和 update 同步到 ~/.hermes/skills/ builtin
optional skills Hermes 仓库 optional-skills/ hermes skills install official/... official
hub skills skills.sh、GitHub、well-known、ClawHub、LobeHub、browse.sh 等 hermes skills install ... trusted/community
plugin skills Hermes 插件提供 plugin:skill 形式加载 取决于插件

5. 标准 Skill 包结构#

推荐结构：

<skill-name>/
├── SKILL.md                 # 必需：主说明文件
├── README.md                # 可选：给人看的安装/交接说明
├── references/              # 可选：长参考资料、API 文档、经验库
│   ├── api.md
│   └── edge-cases.md
├── templates/               # 可选：输出模板、配置模板、提示模板
│   ├── report.md
│   └── config.yaml
├── scripts/                 # 可选：辅助脚本
│   ├── analyze.py
│   └── export.sh
└── assets/                  # 可选：资产文件、示例图片、静态资源
    └── example.json

目录含义：

目录 作用 加载方式
SKILL.md 主流程和核心指令 skill_view(name)
references/ 长文档、细节、API、背景知识 skill_view(name, "references/x.md")
templates/ 输出格式、配置文件、prompt 模板 skill_view(name, "templates/x.md")
scripts/ 可直接执行的辅助脚本 通过 skill 注入的绝对路径运行
assets/ 资产、示例、非文档材料 按路径引用或读取

注意：

  • 支持目录中的 SKILL.md 不会被当作独立 skill 扫描。
  • 扫描器会跳过 .git、.github、.hub、.archive、.venv、node_modules、__pycache__ 等目录。
  • 支持文件建议保持职责清晰，避免把所有内容塞进主 SKILL.md。

6. SKILL.md 文件格式规范#

6.1 最小合法格式#

SKILL.md 必须从 YAML frontmatter 开始，不能有前导空行或 BOM。

---
name: my-skill
description: Brief description of what this skill does
---

# My Skill

## When to Use
- Use when ...

## Procedure
1. Do ...
2. Verify ...

硬性校验规则：

规则 要求
开头 文件必须以 --- 开头
frontmatter 结束 必须有独立 closing --- 行
YAML frontmatter 必须能解析为 YAML mapping
name 必须存在
description 必须存在，最长 1024 字符
body frontmatter 后必须有非空正文
总长度 agent 写入的 SKILL.md 不超过 100,000 字符
支持文件大小 agent 写入的单个支持文件不超过 1 MiB

6.2 推荐 frontmatter#

---
name: my-skill-name
description: Use when <trigger>. <one-line behavior and outcome>.
version: 1.0.0
author: Hermes Agent
license: MIT
platforms: [linux, macos]
metadata:
  hermes:
    tags: [automation, research, cli]
    related_skills: [other-skill]
---

字段说明：

字段 必需 说明
name 是 skill 逻辑名称；建议小写、数字、连字符/下划线/点
description 是 展示在索引和搜索结果中；应写触发条件和用途
version 否 版本号；建议保留
author 否 作者；建议保留
license 否 许可证；建议保留
platforms 否 限制平台：macos、linux、windows
metadata.hermes.tags 否 分类/关键词
metadata.hermes.related_skills 否 相关 skills

6.3 name 规范#

skill_manage 创建 skill 时，名称必须符合：

^[a-z0-9][a-z0-9._-]*$

并且最长 64 字符。

推荐：

github-workflows
test-driven-development
ocr-and-documents
sn-da-excel-workflow

不推荐：

GitHub Workflow      # 有空格和大写
/my-skill            # 绝对路径感
../my-skill          # 路径穿越
my skill!            # 特殊字符

6.4 description 写法#

description 是模型决定是否加载 skill 的主要依据之一，应当：

  • 开头写触发条件：Use when ...
  • 描述任务类型，而不是单次任务。
  • 包含关键工具/领域词。
  • 不超过 1024 字符。

好例子：

description: Use when operating GitHub end-to-end: authentication, repositories, pull requests, code reviews, issues, and CI/CD with gh CLI and git via terminal.

差例子：

description: GitHub stuff.

6.5 platforms 平台限制#

platforms: [macos]
platforms: [linux, macos]
platforms: [windows]

行为：

  • 不匹配当前 OS 时，该 skill 不显示在系统 prompt、skills_list、slash commands 中。
  • 显式 skill_view 加载时也会返回 unsupported。
  • 不写或为空代表所有平台可用。

7. Hermes 专属 metadata 规范#

Hermes 主要读取 metadata.hermes 下的字段。

完整示例：

metadata:
  hermes:
    tags: [web, automation, scraping]
    category: research
    related_skills: [web-access, research-intelligence-workflows]
    requires_toolsets: [web]
    requires_tools: [web_search]
    fallback_for_toolsets: [browser]
    fallback_for_tools: [browser_navigate]
    config:
      - key: myplugin.path
        description: Path to the plugin data directory
        default: ~/myplugin-data
        prompt: Plugin data directory path
    blueprint:
      schedule: "0 9 * * *"
      deliver: origin
      prompt: "Run the daily workflow."
      no_agent: false

7.1 tags / related_skills#

metadata:
  hermes:
    tags: [python, automation]
    related_skills: [github-workflows, test-driven-development]

用途：

  • tags 帮助检索、目录展示、人工理解。
  • related_skills 帮助 agent 或用户发现配套 skill。

7.2 条件激活 requires / fallback#

条件激活控制 skill 是否出现在当前会话的 skill index 中。

metadata:
  hermes:
    requires_toolsets: [web]
    requires_tools: [web_search]
    fallback_for_toolsets: [browser]
    fallback_for_tools: [browser_navigate]

语义：

字段 行为
requires_toolsets 列出的任一 toolset 不可用时隐藏
requires_tools 列出的任一 tool 不可用时隐藏
fallback_for_toolsets 列出的任一 toolset 可用时隐藏；不可用时显示
fallback_for_tools 列出的任一 tool 可用时隐藏；不可用时显示

典型用法：

  • requires_toolsets: [web]：只有联网工具可用时才显示。
  • fallback_for_tools: [web_search]：没有 web_search 工具时才显示替代搜索 skill。

7.3 config：非敏感设置#

Skill 可以声明需要的非敏感配置，存入 config.yaml 的 skills.config 下。

metadata:
  hermes:
    config:
      - key: myplugin.path
        description: Path to the plugin data directory
        default: ~/myplugin-data
        prompt: Plugin data directory path

存储后类似：

skills:
  config:
    myplugin:
      path: ~/myplugin-data

运行时行为：

  • hermes config migrate 可发现并提示补齐未配置项。
  • skill 被 slash command 加载时，Hermes 会把解析后的配置值注入 skill 消息：
[Skill config (from ~/.hermes/config.yaml):
  myplugin.path = /home/user/myplugin-data
]

注意：

  • config 用于路径、偏好、开关等非敏感值。
  • API key、token 这类秘密不要放 config，应放 required environment variables。

7.4 blueprint：skill 兼作自动化蓝图#

Skill 可以声明 metadata.hermes.blueprint，作为可安装的自动化建议。

metadata:
  hermes:
    tags: [blueprint, email]
    blueprint:
      schedule: "0 8 * * *"
      deliver: telegram
      prompt: "Summarize unread email and today's calendar."
      no_agent: false

行为：

  • 安装带 blueprint 的 skill 不会自动创建 cron job。
  • Hermes 会生成 suggested cron job。
  • 用户通过 /suggestions 显式接受后才创建定时任务。

8. 环境变量与凭证文件规范#

8.1 required_environment_variables#

用于声明 API key、token 等秘密。

required_environment_variables:
  - name: TENOR_API_KEY
    prompt: Tenor API key
    help: Get a key from https://developers.google.com/tenor
    required_for: GIF search functionality

字段：

字段 必需 说明
name 是 环境变量名，如 TENOR_API_KEY
prompt 否 本地 CLI 安全输入时展示
help 否 获取 key 的说明或 URL
required_for 否 哪个功能需要它

运行时行为：

  1. skill_view 加载 skill 时读取该声明。
  2. Hermes 检查 ~/.hermes/.env 中是否已有对应变量。
  3. 本地 CLI 可安全提示输入；用户可跳过。
  4. Gateway/消息平台不会在聊天中收集 secret，只给本地配置提示。
  5. 已存在的变量会自动注册为 terminal / execute_code sandbox passthrough。

也就是说，skill 的脚本可以直接访问：

echo "$TENOR_API_KEY"

或 Python：

import os
key = os.environ["TENOR_API_KEY"]

8.2 legacy prerequisites.env_vars#

旧格式仍兼容：

prerequisites:
  env_vars: [API_KEY]
  commands: [curl, jq]

但新 skill 推荐使用 required_environment_variables。

8.3 required_credential_files#

用于 OAuth token、service account JSON、证书等文件型凭证。

required_credential_files:
  - path: google_token.json
    description: Google OAuth2 token created by setup script
  - path: google_client_secret.json
    description: Google OAuth client secret

行为：

  • path 相对 ~/.hermes/。
  • 加载 skill 时检查文件是否存在。
  • 文件存在时可注册给 Docker/Modal 等远程 backend 挂载/同步。
  • 缺失时返回 setup_needed 信息。

9. SKILL.md 正文结构规范#

推荐结构：

# Skill Title

## Overview
一句话说明 skill 解决什么问题，以及它为什么存在。

## When to Use
- 触发条件 1
- 触发条件 2
- Don't use for: 不适用边界

## Quick Reference
| 场景 | 命令/做法 |
|---|---|
| ... | ... |

## Procedure
1. 第一步：检查前置条件。
2. 第二步：执行核心流程。
3. 第三步：处理失败分支。
4. 第四步：验证结果。

## Common Pitfalls
1. 坑点：原因；修复方式。
2. 坑点：原因；修复方式。

## Verification Checklist
- [ ] 结果文件存在
- [ ] 测试通过
- [ ] 用户要求全部满足

写作原则：

  1. 先写最常用流程，再写边缘情况。
  2. 用明确步骤，不写抽象口号。
  3. 命令要可复制运行。
  4. 坑点要写“怎么识别 + 怎么修”。
  5. 验证标准要可执行、可观察。
  6. 对危险操作明确要求确认。
  7. 不要把一次任务的临时细节写进 skill。
  8. 大段参考资料放 references/，主文只保留路由和关键流程。

10. 支持文件的加载机制#

当 skill_view(name) 加载主 skill 时，返回结果里会包含 linked_files。

示例：

{
  "success": true,
  "name": "example-skill",
  "content": "...SKILL.md...",
  "skill_dir": "/root/.hermes/skills/category/example-skill",
  "linked_files": {
    "references": ["references/api.md"],
    "templates": ["templates/report.md"],
    "scripts": ["scripts/analyze.py"],
    "assets": ["assets/schema.json"]
  },
  "usage_hint": "To view linked files, call skill_view(name, file_path) ..."
}

加载支持文件：

skill_view(name="example-skill", file_path="references/api.md")

安全限制：

  • file_path 不能包含 ..。
  • 解析后必须仍在 skill 目录内。
  • 支持文件不存在时，Hermes 会列出可用文件。
  • 二进制文件不能作为文本读取时，会返回 binary 信息。

11. Skill 调用机制#

11.1 自然语言调用#

用户可以直接说：

用 github-workflows 帮我创建 PR

模型会：

  1. 从系统 prompt 的 skill index 看到 github-workflows。
  2. 判断它相关。
  3. 调用 skill_view(name="github-workflows")。
  4. 按加载出的指令执行。

11.2 tools 调用#

可用工具：

工具 作用
skills_list(category?) 列出可用 skill 的 name/description/category
skill_view(name, file_path?) 加载完整 skill 或支持文件
skill_manage(...) 创建、修改、删除 skill

skills_list 返回精简信息；skill_view 才返回完整正文。

11.3 slash command 调用#

每个安装的 skill 会自动生成一个 slash command。

示例：

/github-workflows create a PR for the auth refactor
/excalidraw draw the architecture
/axolotl help me fine-tune a Llama model

调用过程：

  1. Hermes 扫描 skill 目录，生成 /skill-name 命令映射。
  2. 用户输入 slash command。
  3. Hermes 加载 skill 全文。
  4. 构造一条模型可读消息，包含：
    • skill 被调用的说明。
    • 完整 skill 内容。
    • skill 绝对目录。
    • 支持文件列表。
    • 用户在 slash command 后附带的指令。
    • setup/config/runtime note。
  5. agent 根据这条消息继续执行。

slash command 名称规范化：

  • 转小写。
  • 空格和下划线转连字符。
  • 移除非 a-z0-9- 字符。
  • 多个连字符压缩成一个。

11.4 skill command 消息骨架#

slash command 加载后，Hermes 会生成类似内容：

[IMPORTANT: The user has invoked the <skill-name> skill. The full skill content is loaded below.]

<SKILL.md content>

[Skill directory: /abs/path/to/skill]
Resolve any relative paths in this skill ... against that directory ...

[This skill has supporting files:]
- references/api.md -> /abs/path/references/api.md
- scripts/foo.js -> /abs/path/scripts/foo.js

Load any of these with skill_view(...), or run scripts directly by absolute path ...

The user has provided the following instruction alongside the skill invocation: <user instruction>

这个设计的关键点：

  • Skill 内容作为用户消息进入 agent loop，而不是永久修改 system prompt。
  • 这样不会破坏 prompt cache。
  • Memory provider 会尝试只提取用户原始 instruction，避免把整篇 skill 误存进记忆。

11.5 预加载 skill#

CLI 支持：

hermes --skills github-workflows
hermes -s github-workflows,test-driven-development

会话内支持：

/skill github-workflows

12. 系统提示中的 Skill Index 机制#

Hermes 会在系统提示中加入一个 compact skill index。

大致结构：

## Skills (mandatory)
Before replying, scan the skills below. If a skill matches ... MUST load it ...

<available_skills>
  github:
    - github-workflows: Use when operating GitHub end-to-end...
  software-development:
    - test-driven-development: Use when implementing any feature...
</available_skills>

构建流程：

  1. 扫描 ~/.hermes/skills/。
  2. 扫描 skills.external_dirs。
  3. 读取每个 SKILL.md 的 frontmatter 和 description。
  4. 过滤不匹配平台的 skill。
  5. 过滤当前环境不相关的 skill。
  6. 过滤 disabled skill。
  7. 应用 requires/fallback 条件。
  8. 按 category 分组。
  9. 写入系统提示。

缓存机制：

  • 进程内 LRU cache。
  • 磁盘 snapshot .skills_prompt_snapshot.json。
  • cache key 包含：skills dir、external dirs、可用工具、可用 toolsets、平台、disabled 列表、压缩类别等。

这意味着：

  • 新增/删除 skill 后，当前会话未必立刻刷新系统 prompt。
  • slash command 可以 reload skill map，但不一定重建系统 prompt cache。
  • 新会话通常能看到最新 index。

13. Skill 发现与查找规则#

skill_view(name) 的查找策略包括：

  1. 插件 qualified name：plugin:skill。
  2. 分类 fall-through：category:skill 可尝试映射为 category/skill。
  3. 直接路径：skills_dir/name/SKILL.md。
  4. 递归按目录名查找。
  5. 递归按 frontmatter name 查找。
  6. legacy 平铺 Markdown：name.md。

安全限制：

  • name 不能是绝对路径。
  • 不能包含 .. 路径穿越。
  • Windows drive path 也会拒绝，避免误识别为 plugin namespace。
  • 多个候选匹配时，Hermes 拒绝猜测并要求用完整相对路径。

14. Skill 启用、禁用、平台过滤#

Skill 可能因为以下原因不显示：

  1. platforms 不匹配当前系统。
  2. metadata.hermes.requires_* 条件不满足。
  3. metadata.hermes.fallback_for_* 对应工具已可用。
  4. 用户在 skills config 中禁用。
  5. 当前运行环境不匹配 environment tags。
  6. 所在目录被排除扫描。

常用管理命令：

hermes skills list
hermes skills config
hermes tools list
hermes tools enable <toolset>
hermes tools disable <toolset>

如果启用/禁用后不生效：

  • 在 CLI 中开新会话或 /reset。
  • gateway 中重启或 /restart。
  • 新 skill 当前 session 看不到时，通常是 loader/cache 行为，不一定是 bug。

15. skill_manage 写入机制#

skill_manage 是 agent 维护 skill 的工具。

Actions：

action 用途 关键参数
create 新建 skill name, content, optional category
patch 局部替换，推荐 name, old_string, new_string, optional file_path
edit 整体重写 SKILL.md name, content
delete 删除 skill name, optional absorbed_into
write_file 写支持文件 name, file_path, file_content
remove_file 删除支持文件 name, file_path

15.1 create#

skill_manage(
  action="create",
  name="my-skill",
  category="software-development",
  content="""---
name: my-skill
description: Use when ...
version: 1.0.0
author: Hermes Agent
license: MIT
---

# My Skill
...
"""
)

行为：

  1. 校验 name。
  2. 校验 category。
  3. 校验 frontmatter。
  4. 校验内容大小。
  5. 检查是否已有同名 skill。
  6. 创建目录。
  7. 原子写入 SKILL.md。
  8. 如开启 skills.guard_agent_created，执行安全扫描；若阻止则回滚。

15.2 patch#

适合小修小补。

skill_manage(
  action="patch",
  name="my-skill",
  old_string="旧内容",
  new_string="新内容"
)

也可以 patch 支持文件：

skill_manage(
  action="patch",
  name="my-skill",
  file_path="references/api.md",
  old_string="旧内容",
  new_string="新内容"
)

推荐优先用 patch，因为：

  • token 少。
  • 不容易误删大段内容。
  • 更适合维护已有 skill。

15.3 edit#

用于大改，需要提供完整新的 SKILL.md。

风险：

  • 容易丢失原内容。
  • 更消耗 token。
  • 应先读取旧 skill 再完整重写。

15.4 write_file / remove_file#

write_file 只能写以下位置：

references/
templates/
scripts/
assets/
SKILL.md

路径限制：

  • 不允许 ..。
  • 必须在 skill 目录内。
  • 支持文件大小限制 1 MiB。

15.5 delete#

删除有防护：

  • 不能删除 pinned skill。
  • 不能删除 skills root 本身。
  • 不能删除不在已知 skill root 内的路径。
  • 拒绝删除 symlink/junction 指向的目录。

删除时建议传 absorbed_into：

skill_manage(action="delete", name="old-skill", absorbed_into="new-skill")

如果只是清理，无合并目标：

skill_manage(action="delete", name="old-skill", absorbed_into="")

16. 安全机制#

16.1 路径安全#

Hermes 对 skill name 和 file_path 做路径限制：

  • 禁止绝对路径。
  • 禁止 ..。
  • 支持文件必须解析在 skill 目录内。
  • 删除操作必须在已知 skills root 内。

16.2 Prompt injection 提示#

加载 skill 时，Hermes 会扫描一些常见 prompt injection 字符串，例如：

ignore previous instructions
you are now
system prompt:
<system>

发现后会记录 warning，但本地 skill 一般仍会加载。第三方 hub 安装会经过更完整的安全扫描。

16.3 Hub 安装安全扫描#

Hub 安装的第三方 skills 会扫描：

  • 数据外传模式。
  • prompt injection。
  • 破坏性命令。
  • shell injection。
  • 供应链风险。

trust level：

level 来源 策略
builtin Hermes bundled skills 默认可信
official Hermes optional skills 默认可信
trusted 可信 registry/repo 较宽松
community 第三方来源 非 dangerous 可 force；dangerous 不允许

16.4 agent-created skill guard#

配置：

skills:
  guard_agent_created: true

默认关闭。开启后，agent 通过 skill_manage 创建/修改的 skill 会经过 security scan；阻止时会回滚写入。

16.5 write approval#

配置：

skills:
  write_approval: true

开启后，skill_manage 的写操作会先 staged 到：

~/.hermes/pending/skills/

用户通过命令审批：

/skills pending
/skills diff <id>
/skills approve <id>
/skills reject <id>
/skills approval on|off

17.1 模板变量#

Skill 正文中可使用：

${HERMES_SKILL_DIR}
${HERMES_SESSION_ID}

加载时替换：

变量 替换为
${HERMES_SKILL_DIR} 当前 skill 目录绝对路径
${HERMES_SESSION_ID} 当前 session id；没有则保留原样

示例：

Run:

```bash

关闭方式：

```yaml
skills:
  template_vars: false

17.2 inline shell#

Skill 可以写：

Current date: !`date -u +%Y-%m-%d`
Git branch: !`git rev-parse --abbrev-ref HEAD`

但默认关闭。

开启：

skills:
  inline_shell: true
  inline_shell_timeout: 10

行为：

  • 每个 !cmd`` 用 bash 执行。
  • cwd 是 skill 目录。
  • stdout 被内联到 skill 内容中。
  • 输出最多 4000 字符。
  • 超时/失败会插入 [inline-shell error: ...]。

安全注意：

  • inline shell 会在 host 上执行，且加载 skill 时执行。
  • 只应对可信 skill 源开启。

18. Skill Bundles#

Skill bundle 是一份 YAML，把多个 skill 绑定成一个 slash command。

位置：

~/.hermes/skill-bundles/<slug>.yaml

格式：

name: backend-dev
description: Backend feature work — review, test, PR workflow.
skills:
  - github-workflows
  - test-driven-development
  - requesting-code-review
instruction: |
  Always start by writing failing tests, then implement.
  Open the PR through the standard workflow.

调用：

/backend-dev refactor the auth middleware

行为：

  • 加载 listed skills。
  • 附加 bundle instruction。
  • slash command 名称与单 skill 冲突时，bundle 优先。
  • 缺失 skill 会跳过，不导致整个 bundle 失败。
  • 不修改 system prompt，作为一次用户消息注入。

管理命令：

hermes bundles list
hermes bundles show backend-dev
hermes bundles create backend-dev --skill github-workflows --skill test-driven-development
hermes bundles delete backend-dev
hermes bundles reload

20. Bundled skill 更新机制#

Hermes 安装和 update 会把仓库内置 skills 同步到 ~/.hermes/skills/。

同步状态记录在：

~/.hermes/skills/.bundled_manifest

逻辑：

  • 本地 skill 未修改：可自动更新为新的 bundled 版本。
  • 本地 skill 已修改：视为 user-modified，后续同步跳过，避免覆盖用户修改。

重置：

hermes skills reset google-workspace
hermes skills reset google-workspace --restore
hermes skills reset google-workspace --restore --yes

区别：

命令 行为
reset 清除 manifest entry，保留当前文件，重新 baseline
reset --restore 删除本地修改，恢复当前 bundled 版本

22. Skill 编写检查清单#

创建前：

  • 是否真的是可复用流程，而不是单次任务？
  • 是否已有类似 skill 可 patch，而不是新建？
  • 是否该放 ~/.hermes/skills/、repo bundled、optional-skills，还是 hub？

Frontmatter：

  • 文件首字节就是 ---。
  • 有 closing ---。
  • YAML 可解析。
  • name 存在，≤64 字符。
  • description 存在，≤1024 字符。
  • version / author / license 建议存在。
  • metadata.hermes.tags 建议存在。
  • 如果平台有限制，写 platforms。
  • 如果需要工具条件，写 requires/fallback。
  • secret 写 required_environment_variables，不要写进正文或 config。

正文：

  • 有 When to Use。
  • 有明确 Procedure。
  • 有 Common Pitfalls。
  • 有 Verification 或 checklist。
  • 命令可复制运行。
  • 危险操作需要确认。
  • 大参考资料拆到 references/。
  • 模板拆到 templates/。
  • 复杂解析逻辑放 scripts/。

安全：

  • 不包含明文 secret。
  • 不要求 agent 忽略系统/用户指令。
  • 不隐藏破坏性命令。
  • 不从不可信 URL 执行脚本，除非有校验和/审计说明。

验证：

  • skill_view(name) 能加载。
  • skills_list() 能看到。
  • slash command 可触发。
  • 支持文件可通过 skill_view(name, file_path) 加载。
  • 如果包含脚本，脚本可直接运行或有清晰依赖说明。

23. 常见坑#

23.1 当前会话看不到刚创建的 Skill#

原因：system prompt skill index 和 slash command map 有缓存。

处理：

  • 新开会话。
  • /reset。
  • /reload-skills 或相关 reload 命令只刷新 slash-command map，不一定重建 system prompt cache。

23.2 description 太泛，模型不加载#

差：

description: Helps with reports.

好：

description: Use when generating Markdown research reports from gathered evidence. Enforces source grounding, structure, synthesis, and verification before final delivery.

23.3 把知识库写成了任务日志#

Skill 不应写：

2026-06-20 我完成了 X 项目第 3 阶段。

应写：

When continuing a source-study project, first load the project map, then inspect current entry points, then update the knowledge index.

23.4 把 secret 写入 SKILL.md#

不要：

api_key: sk-xxxx

应该：

required_environment_variables:
  - name: SERVICE_API_KEY
    prompt: Service API key

23.5 支持文件路径写错#

不要让 agent 猜相对路径。推荐在 skill 里写：

Run:

```bash
python ${HERMES_SKILL_DIR}/scripts/analyze.py <input>


### 23.6 误用 skill_manage create 创建 repo 内置 skill

`skill_manage(action='create')` 创建到当前 profile 的 `~/.hermes/skills/`。

如果你在 Hermes 源码仓库里开发 bundled skill，应写入：

```text

并提交 git。

23.7 过长 SKILL.md#

超过 100,000 字符会被 agent 写入校验拒绝。即使没超，也不建议过长。

拆分原则：

  • 主流程保留在 SKILL.md。
  • 大表格、API、长示例放 references/。
  • 输出格式放 templates/。
  • 可执行逻辑放 scripts/。

26. 最佳实践总结#

  1. Skill 的 description 决定它是否会被正确召回，必须写清触发条件。
  2. 主 SKILL.md 写流程，细节拆支持文件。
  3. 优先用 patch 维护 skill，少用整体 edit。
  4. 复杂逻辑写脚本，不让 LLM 每次临场重写解析器。
  5. 所有 skill 都应有验证步骤。
  6. 发现 skill 错误或遗漏时，立即 patch；否则 skill 会变成长期负债。
  7. 不把临时进度、PR号、commit SHA、一次性日志写进 skill。
  8. secret 永远放 required_environment_variables / .env，不要进 Markdown 正文。
  9. 新 skill 当前会话不可见时，优先考虑缓存/新会话问题。
  10. Skill 是 agent 能力增长的核心机制：把稳定、可复用、经过验证的做法沉淀进去。
