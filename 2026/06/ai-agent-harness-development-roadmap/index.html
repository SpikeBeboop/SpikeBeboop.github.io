过去一两年，很多人对 AI Agent 的理解还停留在“给大模型一个目标，然后让它自己循环调用工具”。这个说法不能说错，但它太容易把人带偏：真正能落地的 Agent，不是一个 prompt，也不是一个 while loop，而是一套工程系统。

一个可靠的 AI Agent 系统至少要回答这些问题：模型如何接入？上下文如何构建？知识如何检索？记忆如何写入？工具如何授权？任务如何规划？失败如何恢复？多 Agent 如何分工？结果如何评估？线上如何观测？成本如何控制？安全事故如何避免？系统如何部署和回滚？

如果这些问题没有答案，所谓 Agent 很可能只是一个演示视频里跑通的玩具。它可能偶尔表现惊艳，但很难被交给真实用户、真实业务和真实生产环境。

这篇文章整理一条完整的 AI Agent 开发学习路线。它的目标不是教你“用某个框架快速做 Demo”，而是帮助你建立从 LLM API 到 Harness 工程的系统能力：先学会稳定调用模型，再把模型包进一套可执行、可评估、可观测、可恢复、可部署的运行框架里。这里的 Harness，可以理解为围绕 LLM 的工程外骨骼：它负责接入模型、构建上下文、注册工具、保存状态、驱动循环、记录 trace、运行 eval、控制权限和承载部署。

1. 先建立正确地图：AI Agent 不是单点技术#

AI Agent 开发不是一个单独技能，而是一组能力的组合。标题里说“从 LLM API 到 Harness 工程”，重点就在这里：LLM API 只是能力入口，Harness 才是把模型能力变成稳定软件系统的工程层。

可以把它拆成十层：

AI Agent Developer
├── LLM API 基础能力
├── AI 应用产品能力
├── LLM 应用工程能力
├── RAG / 知识 / 上下文能力
├── Memory 能力
├── Tool Calling / MCP 能力
├── Harness / Agent Runtime 能力
├── Subagent / Multi-Agent 编排能力
├── 评估 / 观测 / 安全 / 部署能力
└── 源码学习与架构抽象能力

其中 Harness / Agent Runtime 是中枢层。它不是某个单一框架，而是一组工程职责的集合：

Harness
├── Model Adapter / LLM Gateway
├── Prompt & Context Builder
├── Tool Registry / MCP Adapter
├── Permission Policy
├── Agent Loop / State Machine
├── Planner / Executor / Verifier
├── Memory Manager
├── Event Log / Trace Store
├── Artifact Store
├── Eval Runner
├── Sandbox / Runtime Isolation
└── Deployment & Ops

其中最容易被忽略的是：AI Agent 首先是软件系统，其次才是模型应用。模型负责不确定性推理，系统负责边界、状态、权限、恢复、追踪和验证。

所以学习顺序不应该是“先找最火的 Agent 框架”，而应该是：

LLM API 基础
  -> 工程基础
  -> AI 应用开发
  -> LLM Gateway
  -> RAG / Memory / Context
  -> Tool Runtime / MCP
  -> Harness 核心：Agent Loop / State / Trace
  -> Planning / Verification / Reliability
  -> Subagent / Multi-Agent
  -> Production Harness Runtime
  -> Evaluation / Observability / Safety / Deployment
  -> Source Study

这个顺序的核心逻辑是：先把一个小系统做稳，再增加自主性；先让 Agent 会安全执行，再让它多步规划；先让单 Agent 可追踪，再做多 Agent 并行。

2. 第一阶段：工程基础不是可选项#

很多 AI 应用失败，不是因为模型不够强，而是因为工程底座太弱。

一个 Agent 系统会频繁遇到这些基础问题：

  • 读写文件。
  • 调用 HTTP API。
  • 运行命令。
  • 处理 stdout / stderr / exit code。
  • 持久化任务状态。
  • 管理后台 worker。
  • 记录结构化日志。
  • 做错误分类和重试。
  • 写测试和评测样例。

因此，在接触复杂 Agent 之前，至少要能做出一个基础任务系统：

CLI / API
  -> SQLite / Postgres
  -> task table
  -> worker
  -> logs
  -> tests

一个很好的入门练习是实现命令行任务管理器：

agent-task add "read paper"
agent-task list
agent-task done <id>
agent-task log <id> "note"

然后把它封装成 HTTP API：

POST /tasks
GET /tasks
PATCH /tasks/:id
POST /tasks/:id/logs

再加一个后台 worker，从数据库读取 pending 任务，执行、记录结果、失败重试。

这个练习看起来和 AI 无关，但它会直接迁移到 Agent Runtime：用户目标是 task，Planner 会生成 task，Executor 会消费 task，工具执行会写 logs，失败恢复会依赖状态机。

如果你不能稳定实现这些基础设施，就不要急着做多 Agent。

3. 第二阶段：LLM API 基础，先把模型调用吃透#

在谈 Agent 之前，先把 LLM API 当成一个不稳定但强大的外部依赖来处理。很多 Agent 系统的问题，本质上是最底层模型调用没有工程化：没有 timeout，没有 retry，没有 token 统计，没有 raw response，没有结构化输出校验，也没有 provider 差异抽象。

这一阶段要掌握的不是“如何发一个请求”，而是如何把模型请求变成可控的系统能力。

最小学习范围包括：

HTTP request / response
authentication
chat completions / responses API
messages / input schema
system / user / assistant roles
streaming / SSE
JSON mode / structured output
tool call format
usage / token accounting
rate limit / retry / timeout
error normalization

一个最小 LLM API 实验应该包括：

  1. 调用一个模型完成普通问答。
  2. 调用一个模型输出固定 JSON schema。
  3. 开启 streaming，并把 chunk 合并成完整输出。
  4. 记录 latency、token usage、status、error type。
  5. 故意触发超时、限流、JSON 解析失败，并做错误分类。

建议从第一天就保存调用记录：

LLMCall
- id
- provider
- model
- request_hash
- prompt_version
- input_tokens
- output_tokens
- latency_ms
- status
- error_type
- raw_request_path
- raw_response_path

这样后续你才有能力回答：哪个 prompt 变贵了？哪个模型变慢了？哪个 provider 出错最多？结构化输出失败率是多少？

这一阶段的目标是：不要把 LLM 当魔法函数，而要把它当成一个需要封装、观察、降级和评估的远程服务。

4. 第三阶段：AI 应用开发，不要把一切做成聊天框#

AI 应用不等于聊天机器人。

真实 AI 应用至少有三种形态：

3.1 Copilot 型应用#

用户仍是主操作者，AI 做辅助：总结、补全、建议、检索、草稿。

典型例子：编程助手、写作助手、客服坐席助手、数据分析助手。

关键是低延迟、可编辑、可撤销、人类确认。

3.2 Workflow 型应用#

业务流程由代码控制，LLM 只负责其中某些节点。

比如文档审核、简历筛选、票据处理、报告流水线。

这类应用不一定需要 Agent loop。流程稳定时，用规则和 workflow 控制通常更可靠。

3.3 Agent 型应用#

用户给出目标，系统自动规划、调用工具、执行多步任务。

比如自动研究助手、自动代码修复助手、工单处理 Agent。

这类应用必须有任务状态、工具权限、trace、失败恢复和最终验证。

做 AI 应用前，先写清楚：

用户是谁？
用户原来怎么完成任务？
痛点是什么？
AI 介入哪个环节？
AI 输出给谁使用？
失败成本是什么？
哪些操作必须由用户确认？
用户如何纠错和反馈？

不要默认把所有需求都塞进聊天框。聊天框只是入口，不是产品结构。

5. 第四阶段：LLM Gateway，把模型调用变成工程模块#

很多项目最初都会这样写：

业务代码 -> 直接调用某个模型 SDK

这在 Demo 里没问题，但很快会失控。你会遇到不同 provider、不同协议、不同 streaming 格式、不同 tool calling 格式、不同错误码、不同 token 计费方式。

所以要尽早建立 LLM Gateway：

Application
  -> LLM Gateway
      -> Provider Adapter A
      -> Provider Adapter B
      -> Provider Adapter C

一个合格的 LLM Gateway 至少要处理：

  • 多 provider 接入。
  • chat completions / responses 等协议差异。
  • 非流式和流式输出。
  • JSON schema / structured output。
  • tool call 格式归一化。
  • prompt versioning。
  • token / cost / latency 记录。
  • timeout / retry / fallback。
  • raw response 保存。

推荐抽象：

LLMRequest
- model
- messages / input
- system instruction
- tools
- response_format
- temperature
- max_tokens
- stream
- metadata

LLMResponse
- text
- structured_output
- tool_calls
- usage
- latency
- provider
- raw_response

结构化输出尤其重要。Agent 系统里大量决策都不是自由文本，而是结构化数据：任务计划、工具参数、路由决策、错误分类、评估结果。

当 JSON 解析失败时，不应该系统崩溃，而应该进入 repair loop：把 validation error、原始输出和目标 schema 一起反馈给模型，最多修复 1-2 次，仍失败就返回结构化错误。

6. 第五阶段：RAG 与 Context Engineering#

Agent 不能只依赖模型参数记忆。真实系统必须能使用外部知识：文档、代码、项目规范、API 手册、历史记录、用户偏好。

RAG 的重点不是“把文档塞进向量库”，而是构建一条可诊断的知识链路：

Document parsing
  -> Chunking
  -> Embedding / Indexing
  -> Retrieval
  -> Reranking
  -> Context assembly
  -> Answer generation
  -> Citation verification

RAG 失败时，要能分类：

  • ingestion failure：文档没导入或解析错。
  • chunking failure：答案被切断。
  • retrieval failure：正确 chunk 没召回。
  • ranking failure：召回了但排太低。
  • context assembly failure：上下文噪声太多。
  • generation failure：上下文正确但模型答错。
  • citation failure：引用不支持结论。

不同错误对应不同修复方式。不能一遇到 RAG 不准就盲目换 embedding 模型。

一个成熟的检索流程通常是：

query rewrite
-> vector search top 30
-> keyword search top 30
-> metadata filter
-> merge and dedupe
-> rerank top 10
-> context compression top 5
-> answer with citations

尤其在代码库、API 文档、错误日志场景里，关键词检索很重要。函数名、错误码、配置项、类名往往不是向量检索最擅长的内容。

Context Engineering 的目标是：在有限上下文窗口里放入最有用、最可信、最相关的信息。

优先级大致是：

当前用户目标
-> 安全和权限规则
-> 当前任务状态
-> 输出 schema
-> 相关工具说明
-> 检索证据
-> 长期记忆
-> 最近对话摘要
-> 历史相似任务经验

低优先级内容不能挤掉高优先级内容。上下文不是越多越好，而是要被选择、压缩、排序和验证。

7. 第六阶段：Memory，不要把记忆做成垃圾桶#

Agent 的 memory 经常被误解为“把所有历史对话总结一下”。这很危险。

记忆至少要分层：

6.1 Working Memory#

一次任务内的临时状态：当前目标、计划、中间观察、工具结果、未解决问题。

它属于当前 run，不默认进入长期记忆。

6.2 Episodic Memory#

历史任务经验：某类任务怎么做、哪些工具组合有效、失败案例。

适合相似任务召回。

6.3 Semantic Memory#

稳定事实：用户偏好、项目路径、API 约定、团队规范、环境事实。

这类可以长期保存，但要有来源、scope、confidence、更新时间。

6.4 Procedural Memory#

可复用流程：如何 debug、如何发布、如何 code review、如何跑 eval。

这类更适合沉淀为 skill、playbook、workflow，而不是普通记忆。

长期记忆写入前，应该经过：

candidate extraction
-> classification
-> stability check
-> conflict check
-> sensitivity check
-> deduplication
-> write / update / reject

不要写入：临时任务进度、PR 编号、今天修了什么、临时 token、未验证的模型猜测。

适合写入：长期偏好、项目稳定路径、API 约定、环境事实、稳定工作流。

记忆系统的关键不是“记得多”，而是“记得准、记得有边界、需要时能召回、不需要时不污染上下文”。

8. 第七阶段：Tool Calling 与 Tool Runtime#

工具调用是 Agent 和普通聊天机器人的分界线。Agent 不只是回答，它会读文件、查网页、调用 API、运行命令、修改状态。

这也是风险最大的地方。

不要把函数直接暴露给模型。你需要 Tool Runtime：

Agent
  -> Tool Registry
  -> Validator
  -> Permission Policy
  -> Executor
  -> Tool Result Normalizer
  -> ToolRun Log

每个工具都应该有：

Tool
- name
- description
- input_schema
- output_schema
- risk_level
- permissions_required
- timeout
- retry_policy
- examples

工具按风险分层：

等级 示例 策略
low 读取公开网页、读取安全目录 自动允许
medium 写临时文件、运行测试 可自动但记录日志
high 修改项目文件、安装依赖、调用外部 API 需要策略允许或确认
critical 删除数据、发消息、生产部署、支付 必须显式确认

工具结果也不要只返回字符串，建议结构化：

{
  "status": "success|error",
  "summary": "...",
  "data": {},
  "stdout": "...",
  "stderr": "...",
  "error_type": null,
  "truncated": false,
  "artifacts": []
}

错误要归一化：

VALIDATION_ERROR
PERMISSION_DENIED
TIMEOUT
NOT_FOUND
RATE_LIMITED
TRANSIENT_ERROR
FATAL_ERROR

模型看到错误类型后，才能决定是修参数、换工具、重试、请求用户确认，还是停止。

9. 第八阶段：MCP，把工具生态标准化#

MCP（Model Context Protocol）解决的是工具生态标准化问题。

传统工具调用的问题是：每个应用都要自己接工具，schema、认证、错误格式、连接生命周期都不统一。MCP 把外部能力标准化为 server、tools、resources、prompts 和 transports。

基本结构：

Agent Runtime as MCP Client
  -> connect MCP Server
  -> list tools/resources/prompts
  -> register tools into Tool Registry
  -> call tool
  -> normalize result
  -> audit log

关键点：MCP 不替代 Tool Runtime。MCP 工具仍然必须经过本地权限、审计、超时、输出截断和错误归一化。

推荐架构：

Agent
  -> Tool Registry
  -> Permission Policy
  -> Tool Executor
      -> Native Tool
      -> MCP Tool Adapter
          -> MCP Server

如果 MCP server 很多，还可以加 MCP Gateway：

Agent Runtime
  -> MCP Gateway
      -> server registry
      -> auth / policy
      -> tool discovery cache
      -> call routing
      -> audit log
      -> rate limit
      -> MCP Servers

MCP 学习重点：

  • server / client / tool / resource / prompt / transport。
  • stdio 与 HTTP transport。
  • 工具发现和命名空间。
  • 凭据隔离，不把全部环境变量传给 server。
  • server healthcheck。
  • schema diff。
  • sampling 安全。
  • 不可信 server 的权限限制。

一个很好的练习是实现 notes MCP server：

tools:
- search_notes(query)
- read_note(id)
- create_note(title, content)

resources:
- notes://index
- notes://note/{id}

然后把它接入自己的 Tool Registry。

10. 第九阶段：Harness 工程，把能力装进运行框架#

前面的 LLM Gateway、RAG、Memory、Tool Runtime、MCP 都是能力模块。Harness 工程要做的事，是把这些能力装进一个统一运行框架，让 Agent 不只是“能调用”，而是“能被调度、能被追踪、能被恢复、能被评估”。

可以把 Harness 理解为 Agent 的执行容器。它负责在每一次任务中维护：

Task
-> Run
-> Plan
-> Step
-> LLMCall
-> ToolRun
-> Observation
-> Artifact
-> Verification
-> Final Result

一个最小 Harness 应该提供这些接口：

Harness.start(task)
Harness.step(run_id)
Harness.pause(run_id)
Harness.resume(run_id)
Harness.cancel(run_id)
Harness.get_trace(run_id)
Harness.get_artifacts(run_id)
Harness.run_eval(eval_case_id)

它的核心数据结构通常包括：

Run
- id
- user_goal
- status
- plan
- current_step
- budget
- created_at
- updated_at

Step
- id
- run_id
- type
- input
- output
- status
- evidence
- error_type

ToolRun
- id
- step_id
- tool_name
- input
- output
- permission_decision
- latency_ms

Artifact
- id
- run_id
- type
- path_or_uri
- summary
- checksum

TraceEvent
- id
- run_id
- event_type
- payload
- timestamp

为什么需要 Harness？因为单独的 LLM API 不知道任务是否完成，单独的 tool calling 不知道权限边界，单独的 RAG 不知道引用是否被最终答案正确使用，单独的 memory 不知道什么该长期保存。Harness 把这些能力统一到一个任务生命周期里。

学习 Harness 工程时，建议先实现一个极小版本：

  1. 接收用户任务，创建 Run。
  2. 调用 Planner 生成结构化计划。
  3. 逐 Step 执行。
  4. 每次 LLM call 和 ToolRun 都写 trace。
  5. 每个 Step 写 evidence。
  6. 结束前运行 Verifier。
  7. 产物进入 Artifact Store。
  8. Eval Runner 可以复用同一套执行入口。

这个阶段完成后，你的 Agent 才开始从“脚本”变成“系统”。

11. 第十阶段：Harness 核心：Agent Loop，不要只写 while 循环#

最简 Agent loop 是：

Observe -> Think / Plan -> Act -> Record -> Verify -> Continue or Stop

工程化版本应该是：

load state
-> build context
-> decide next action
-> validate action
-> execute tool or ask user
-> persist observation
-> update plan/state
-> check stop condition
-> verify final output

Agent loop 有多种形态。

ReAct Loop#

适合探索性任务：

Thought -> Action -> Observation -> Thought -> ...

优点是简单通用；缺点是容易发散、成本不可控、难评估。

Plan-and-Execute Loop#

适合开发、研究、文档处理：

Plan -> Execute Step 1 -> Update -> Execute Step 2 -> Verify

State Machine Loop#

适合生产系统：

created -> planning -> executing -> waiting_for_user -> verifying -> completed
                                      -> failed
                                      -> cancelled

Workflow + Agent Hybrid#

适合企业应用：确定性流程由代码控制，困难节点交给 LLM。

可靠 Agent 必须有 stop conditions：

  • success criteria satisfied
  • max steps reached
  • max tool calls reached
  • max cost reached
  • max runtime reached
  • repeated action detected
  • repeated failure detected
  • permission blocked
  • user confirmation required
  • no progress detected

没有 stop condition 的 Agent，只是一个有预算风险的循环。

12. 第十一阶段：Planning、Reflection 与 Verification#

规划不是一次性输出漂亮清单，而是系统可执行的结构化计划。

推荐 Plan Schema：

{
  "goal": "...",
  "assumptions": ["..."],
  "steps": [
    {
      "id": "s1",
      "title": "...",
      "type": "research|code|test|review|deploy|ask_user",
      "depends_on": [],
      "risk": "low|medium|high|critical",
      "can_delegate": true,
      "expected_artifact": "...",
      "verification": "..."
    }
  ],
  "stop_conditions": ["..."],
  "success_criteria": ["..."]
}

规划模式包括：

  • Checklist Planner：适合发布、检查、报告生成。
  • DAG Planner：适合有依赖和并行的复杂任务。
  • Hypothesis Planner：适合 debugging 和研究。
  • Risk-first Planner：适合有副作用任务。

Reflection 也不能只是模型自言自语。它应该是结构化检查：

这一步是否产生预期观察？
是否出现新约束？
是否需要重试、换工具或重规划？
是否消耗超过预算？

Verification 尽量使用确定性手段：

  • tests pass
  • file exists
  • schema validates
  • HTTP status 200
  • diff matches intent
  • eval threshold reached

LLM-as-judge 可以用于语义质量判断，但不能替代高风险决策和确定性验证。

13. 第十二阶段：Subagent 与 Multi-Agent#

多 Agent 的价值不是角色扮演，而是任务隔离、并行探索、互相审查和结构化合并。

适合多 Agent 的任务：

  • 可自然拆成多个独立子任务。
  • 需要并行搜索、并行编码、并行审查。
  • 需要不同视角：实现者、测试者、审查者、研究者。
  • 任务长，上下文多，单 Agent 容易遗忘。

不适合多 Agent 的任务：简单问答、强顺序依赖任务、目标不清任务、高风险副作用动作。

标准多 Agent 结构：

Orchestrator Agent
  -> Task Decomposer
  -> Worker Agents
  -> Reviewer Agents
  -> Merger / Synthesizer
  -> Final Verifier

Subagent handoff 必须明确：

- goal
- background context
- exact files or sources
- constraints
- allowed operations
- forbidden operations
- expected output format
- verification required
- deadline / step budget

Worker 返回也必须结构化：

- what was done
- evidence or paths
- verification result
- unresolved issues
- confidence
- recommended next action

多 Agent 输出冲突时，不能强行糊成一个结论。合并规则应该优先真实工具验证结果、源码证据、可复现命令输出，并显式列出冲突点。

14. 第十三阶段：生产级 Harness Runtime#

一个生产级 Agent Runtime 至少包含：

task intake
run/session/thread model
state machine
context builder
llm gateway
tool registry
permission policy
mcp manager
memory manager
planner
executor loop
subagent scheduler
event bus / queue
artifact store
trace store
eval harness
observability
deployment and ops

核心数据模型：

Session：用户会话或工作区会话
Thread：对话或任务上下文线索
Run：一次 Agent 执行
Step：Run 中的一个决策或动作
ToolRun：一次工具调用
Artifact：文件、报告、代码补丁、图表等产物
Event：运行过程中的事件流

长任务不能阻塞 API 请求。典型架构是：

API receives task -> enqueue run -> worker executes -> event stream updates UI

需要支持：retry、cancel、pause/resume、worker heartbeat、dead letter queue、priority、concurrency limit、workspace quota。

Artifact Store 也很重要。Agent 的产物不应该只存在聊天记录里，而应该保存为：报告、代码 diff、CSV/JSON、截图、日志、eval report。

15. 第十四阶段：评估、观测、安全#

生产级 Agent 不能靠“看起来不错”上线。

Agent 评估要覆盖最终答案和中间过程：

final answer
plan
tool trajectory
intermediate observations
artifacts
cost
latency
safety decisions

Eval 分层：

  • Unit Eval：评估单个 prompt、tool 参数、memory classifier、retriever。
  • Integration Eval：评估 RAG answer、tool error recovery、MCP bridge、planning + execution。
  • End-to-End Eval：评估完整 coding、research、document、data analysis 任务。
  • Safety Eval：评估 prompt injection、unsafe shell、secret exfiltration、unauthorized action。

Trace 是 Agent 的黑匣子记录。它应该能串起：

user task -> plan -> llm calls -> tool calls -> artifacts -> final verification

Observability 至少包含：

run_success_rate
avg_steps_per_run
tool_failure_rate
model_latency
token_usage
cost_per_task_type
eval_pass_rate
human_intervention_rate
permission_denial_rate

安全重点：

  • 外部网页、PDF、GitHub issue、tool output 都是不可信输入。
  • 不要把外部内容里的指令提升为系统规则。
  • secrets 不进 prompt，不进日志，不进错误消息。
  • 高风险工具必须有权限和确认。
  • sandbox 执行代码和 shell。
  • 不同 workspace 的 RAG 和 memory 不能串扰。

发布门禁应该包括：unit tests、core eval subset、safety smoke eval、golden trace diff、成本和延迟阈值。

16. 第十五阶段：部署、扩展与成本控制#

Agent 系统比普通 Web 服务复杂，因为它有长任务、模型调用、工具执行、文件产物、权限和外部 API。

部署成熟度可以分四层：

Stage 0：本地脚本#

单进程、SQLite / JSONL、本地文件。适合学习和个人工具。

Stage 1：本地服务#

API server、本地 worker、SQLite、local artifact directory、简单 UI / CLI。适合个人工作台。

Stage 2：小规模服务#

Postgres、Redis queue、object storage、vector database、worker pool、Docker Compose。适合团队使用。

Stage 3：生产平台#

容器平台、worker autoscaling、sandbox pool、secrets manager、集中式 logs/metrics/traces、CI eval gate、backup/migration。

成本控制要从一开始考虑：

  • 每个 workspace 设置预算。
  • 每个 run 设置 max_cost。
  • 小任务用便宜模型。
  • 复杂规划和最终综合用强模型。
  • 长上下文先压缩。
  • 缓存 deterministic 子结果。
  • eval 分层运行，不每次 full suite。

模型路由可以是：

classification -> cheap model
structured extraction -> medium model
complex planning -> strong model
final synthesis -> strong model
judge eval -> separate judge model

不要所有任务都用最贵模型，也不要为了省钱牺牲关键步骤的可靠性。

17. 第十六阶段：源码学习路线#

AI Agent 框架变化很快。长期能力不是记住框架名，而是能从源码中还原设计。

源码阅读不要从 README 顺着读，而要带问题：

用户任务从哪里进入？
模型调用在哪里？
工具 schema 在哪里定义？
状态如何保存？
loop 如何停止？
trace 如何记录？
eval 如何运行？
权限如何控制？

建议按层研究：

  1. LLM 应用框架：LangChain、LlamaIndex、Haystack、DSPy。
  2. Agent Graph / Workflow：LangGraph、Semantic Kernel planners。
  3. 多 Agent 框架：AutoGen、CrewAI、MetaGPT 类项目。
  4. 编码 Agent：Aider、OpenHands、Continue、Cline / Roo Code、SWE-agent。
  5. MCP 生态：MCP SDK、filesystem server、GitHub server、database server、browser server。
  6. 评测框架：SWE-bench harness、promptfoo、RAGAS、DeepEval。

每读一个项目，至少输出：

01-positioning.md
02-architecture.md
03-execution-flow.md
04-data-model.md
05-tooling-and-mcp.md
06-memory-context-rag.md
07-agent-loop-planning.md
08-eval-observability.md
09-security-deployment.md
10-design-lessons-and-replica.md

读源码的最终目标是复刻最小设计点，比如 tool registry、state graph、retriever interface、MCP adapter、eval runner、patch applier、trace viewer。

只看懂不复刻，学习效果会很浅。

19. 最终 Capstone：生产级 AI Agent Harness Workbench#

最终项目不建议做“一个聊天机器人”，而应该做一个 Agent Workbench：Runtime + 应用壳层 + 工具生态 + 评测系统。

它至少支持四类场景：

  1. 源码理解：读取仓库，分析入口、模块、数据流、扩展点。
  2. 代码修改：定位 bug，修改文件，运行测试，输出 diff 和验证结果。
  3. 资料研究：拆分问题，并行搜索，合成带来源报告。
  4. 知识库问答：基于私有文档回答，不确定时拒绝编造。

模块结构：

Workbench
├── App Layer
├── Runtime Layer
├── Context Layer
├── Tool Layer
├── Model Layer
├── Quality Layer
└── Storage Layer

作品集必须包含：

  • 架构图。
  • 运行截图或演示视频。
  • 三个典型 trace。
  • eval 报告。
  • 失败案例分析。
  • 安全边界说明。
  • 部署说明。
  • 下一步路线。

完成定义不是“功能很多”，而是：能跑、能解释、能评测、能追踪、能恢复、能部署、能说明不能做什么。

20. 常见反模式#

最后列几个常见坑。

反模式 1：聊天框吞天下#

所有功能都塞进一个 chat prompt。结果是状态不可控、权限不可控、难评测。

反模式 2：无 trace 调 prompt#

效果不好就改 prompt，但没有样例和 trace。结果是越改越玄学。

反模式 3：工具无权限#

模型能直接调用 shell、数据库、消息发送。结果是安全事故迟早发生。

反模式 4：RAG 无引用#

回答看起来对，但无法追溯证据。结果是用户无法信任，错误难排查。

反模式 5：多 Agent 角色扮演#

多个 Agent 互相聊天，但没有独立任务、证据和合并机制。结果是成本增加，质量不一定提高。

反模式 6：没有 eval 的上线#

靠几次演示判断质量。结果是回归不可见。

反模式 7：把 memory 当垃圾桶#

所有历史对话都写进长期记忆。结果是上下文污染和隐私风险。

反模式 8：过早框架化#

还没理解 run / step / tool，就引入复杂框架。结果是出了问题不知道去哪查。

结语：Agent 的核心不是“自主”，而是“可控的自主”#

AI Agent 的吸引力在于它能把 LLM 从回答问题推进到执行任务。但越是能执行，越需要工程约束。

真正可用的 Agent，不是更长的 prompt，而是一个有状态、有权限、有工具、有记忆、有评估、有观测、有部署、有安全边界的软件系统。

学习 AI Agent 开发，最好的路线不是追逐框架，而是逐层搭建自己的理解：

LLM API
-> AI 应用场景
-> LLM Gateway
-> RAG / Memory / Context
-> Tool Runtime / MCP
-> Harness 工程
-> Agent Loop / Planning
-> Subagent / Multi-Agent
-> Production Harness Runtime
-> Eval / Observability / Safety / Deployment
-> Source Study

当你能从零实现一个最小 Runtime，再读懂主流框架为什么这样设计，你才真正进入了 AI Agent 开发的核心区域。
